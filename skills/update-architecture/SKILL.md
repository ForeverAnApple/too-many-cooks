---
name: update-architecture
description: Generate or refresh ARCHITECTURE.md from the current codebase structure
---

Generate or update the `ARCHITECTURE.md` file at the workspace root. This file serves as a cached snapshot of the project structure so that agents (especially `@explorer`) don't need to re-explore the entire codebase on every task.

## When to Run
- After creating a new project or cloning a repo that lacks ARCHITECTURE.md
- After significant structural changes (new modules, major refactors, dependency changes)
- When the orchestrator notices ARCHITECTURE.md is stale or missing

## Process

### 1. Quick Scan
Use `glob` and `read` to understand the project. **Do not read every file.** Focus on:
- Top-level directory listing (1 level deep)
- Package manifest (`package.json`, `pyproject.toml`, `Cargo.toml`, `go.mod`, etc.)
- Entry points (`src/index.*`, `src/main.*`, `app.*`, etc.)
- Config files (`.env.example`, `tsconfig.json`, `next.config.*`, `docker-compose.*`, etc.)
- README.md (first 50 lines for project description)

### 2. Identify Key Modules
For each top-level directory that contains source code, read its index/entry file (if any) to understand its purpose. **Do not recurse deeply** — 1-2 levels is enough.

### 3. Write ARCHITECTURE.md
Write the file to the workspace root using this structure:

```markdown
# Architecture

> Auto-generated by `update-architecture` skill. Last updated: [DATE]

## Overview
[1-3 sentences: what this project is, what it does]

## Tech Stack
- **Language**: [e.g., TypeScript, Python, Rust]
- **Framework**: [e.g., Next.js, FastAPI, Axum]
- **Database**: [if applicable]
- **Key Dependencies**: [3-5 most important deps from manifest]

## Project Structure
```
[top-level tree, 2 levels deep max]
```

## Key Modules
| Directory | Purpose |
|-----------|---------|
| `src/api/` | REST API routes and handlers |
| `src/lib/` | Shared utilities and helpers |
| ... | ... |

## Entry Points
- **Main**: `src/index.ts` — Application bootstrap
- **Config**: `src/config.ts` — Environment and app configuration
- [other important entry points]

## Patterns & Conventions
- [e.g., "API routes follow REST conventions in `src/api/[resource]/route.ts`"]
- [e.g., "State management uses Zustand stores in `src/stores/`"]
- [e.g., "Tests are colocated with source files as `*.test.ts`"]
```

### 4. Gitignore Check
ARCHITECTURE.md **should be committed** — it's useful for humans too. Do NOT add it to .gitignore.

## Rules
- Keep it under 80 lines. This is a quick reference, not documentation.
- Use relative paths from the workspace root.
- Don't list every file — focus on directories and key entry points.
- If the project already has an ARCHITECTURE.md, update it in place rather than rewriting from scratch (preserve any human-written sections).
- Include the "Last updated" date so staleness is visible.
